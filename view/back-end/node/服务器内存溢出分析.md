# 内存“溢出”排查与治理经验（weibo-spider：spider/downloader + pm2）

## 背景与现象

> 服务形态：Node 常驻服务（pm2 守护），spider 负责拉取微博数据并入库/入队列，downloader 负责从队列下载图片并上传七牛。

## 现象：
1. pm2 监控的 进程内存（RSS） 会在某些任务后明显抬升（例如从 100MB 级别抬到 200MB+，甚至接近 300MB）。
2. 但 V8 的 Heap 指标（Heap Size/Used Heap/Heap Usage）基本稳定 或可回落。
3. 有时任务结束后 external/arrayBuffers 会回落到很低，但 RSS 仍长期保持在高位。

## 一、先区分：是不是“真正的内存泄漏”

> 结论：本次主要是 堆外内存峰值 + allocator 保留导致 RSS 不回落，不是典型的 JS 堆泄漏。

关键判据
1. 如果是 JS 堆泄漏：
* heapUsed 在多轮任务后持续上升，强制 GC 也难以回落；
* 常伴随对象数量随轮次线性增长（全局数组/Map、未释放闭包、监听器无限注册等）。

2. 如果是堆外/系统层导致的 RSS 抬升（本次情况）：
* heapUsed 强制 GC 后能明显回落；
* external/arrayBuffers 可能在任务期出现峰值，但后续会回落；
* handles 数量稳定（不随轮次增长）；
* RSS 不回落更像 allocator/OS 的“保留以复用/碎片化”，而非代码持有引用。

## 二、诊断方法（可复用的排查模板）

### 在任务关键节点打印：

* rss：pm2 “memory” 主要来源
* heapUsed：JS 堆使用
* external / arrayBuffers：Buffer/ArrayBuffer 相关堆外
* handles：活跃句柄数（粗定位 socket/定时器/流等）

* 并增加“可开关强制 GC”验证：
  环境变量：MEM_DIAG_GC=1
  启动参数：--expose-gc
  在任务结束后打印：
  gc_before
  调用 global.gc()
  gc_after

### 如何解读：

* heapUsed 在 gc_after 明显下降 ⇒ 非 JS 堆泄漏。
* external/arrayBuffers 任务期高、后续回落 ⇒ 临时 Buffer 峰值（下载/上传/SDK 相关）。
* rss 在 gc_after 仍不下降 ⇒ allocator/OS 保留页（常见于长跑网络服务）。

### 三、downloader 侧经验与优化点
* 1）避免定时器/任务叠加
  统一调度，避免 setInterval 内叠加 setTimeout 造成未触发的任务积压。
  增加 is_getting_queue / is_downloading 锁，避免并发触发多次 getQueue()。
* 2）下载端（axios stream）资源治理
  * 对 stream 做完整生命周期清理：end/close/error 都解绑监听并释放引用。
  * 在 error 路径主动 destroy()。
  * 使用 有上限的 keep-alive agent（限制 maxSockets/maxFreeSockets），降低 socket/缓冲过度增长概率。
  * 增加 hard timeout + Abort，处理“卡住但不触发 timeout”的边界。
* 3）认识 RSS 的“平台效应”
  external/arrayBuffers 即便回落，RSS 也可能长期不回落，这是常驻服务的常见表现。
* 4）降低并发（如 BATCH_SIZE）
  降低并发（如 BATCH_SIZE）能 降低峰值概率与峰值高度，但不保证 RSS 回到低位；更多是减缓/降低“平台高度”。

### 四、spider 侧经验：并发导致的 RSS 暴涨

> spider 进程一次冲到 226M 的高风险点：

* 启动时可能触发 重复 run() 并发执行；
* data50.forEach(saveData) 不 await，导致一次并发 50 个保存任务；
* saveData 在单图场景会下载图片 arraybuffer 计算 ratio，并发时 Buffer 峰值极大；
* 网络错误时立即递归重试，可能形成 tight loop。

> 治理策略：

* 单实例运行锁（避免并发 run）
* 串行/限并发保存（for...of await）
* 失败退避（比如 30s 后重试）
* 统一调度器（clearTimeout 再 setTimeout）

### 五、工程上“最终兜底”

> 对于 RSS 不回落这种 allocator/系统层行为，最有效的生产手段是：

- --max-memory-restart 350M（按峰值留 buffer 设阈值）
```bash
pm2 delete download || true
MEM_DIAG_GC=1 pm2 start /home/app/backend/weibo-spider/src/downloader/index-download.js \
  --name download \
  --node-args="--expose-gc" \
  --max-memory-restart 350M \
  --watch
pm2 save
pm2 startup
```
- [ubuntu 服务器上，使用 jemalloc 作为内存分配器，以优化内存使用效率](/view/service/ubuntu/RSS碎片内存优化.md)

### 六、关于升级 Node（18.19 → 新版本）

> 升级 Node（尤其是升到最新 18 LTS patch 或 20 LTS）可能改善：

V8/GC/Buffer/网络栈的一些内存管理与 bug
但对“RSS 不回落”这类 allocator 行为，升级 不保证根治。
最可靠的收益评估：升级前后对比
after_task_finished_gc_after 的 rss 基线
external/arrayBuffers 峰值幅度与频率

### 七、本次结论（给未来复盘的一句话）

> Heap 不涨 + 强制 GC heapUsed 可回落 + handles 不增长 ⇒ 不是 JS 泄漏。
> external/arrayBuffers 偶发峰值 ⇒ 下载/上传链路的堆外 Buffer 峰值是主因。
> RSS 高位不回落 ⇒ allocator/OS 保留与碎片化是常见机制，靠“pm2 内存上限重启 + jemalloc 内存碎片优化”最稳。
